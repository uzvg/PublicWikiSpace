在实现tiddlywiki的大模型助手Jarvis的时候，遇到一个问题，即如何拼凑出质量更高的context文本？哪种结构更好，易读性更高？

- 一种是使用`wikify`组件，效果可见👉[[wikify test]]
- 另一种是原生的wikitext文本。

这是一个非常核心的问题，在将 TiddlyWiki 与大语言模型（LLM）集成时，**你如何表达内容的结构，会极大影响模型的理解效果**。下面是对你提出的两种方案的比较分析：

---

## 🧩 方案 1：wikify 渲染后的纯文本（无语法结构）

### ✅ 优点：

* **简洁直接**，适合用作摘要、简单上下文、纯语义处理。
* 不需要大模型学习 TiddlyWiki 的语法规则，**降低了上下文噪音**。
* 更类似于普通文章，容易转化为 GPT 等模型已经训练过的文档形式。

### ❌ 缺点：

* **丢失了结构信息**：如标题、层级、链接、列表等在渲染后都被“平铺”，失去了“上下文语境”。
* 对于结构敏感的任务（如摘要、提纲、重组、分类），**模型很难准确还原上下文结构**。
* 诸如“链接”、“引用”、“embed”等高级语义几乎完全丢失。

---

## 🧩 方案 2：原始 wikitext 源代码（保留结构语法）

### ✅ 优点：

* **结构保留完好**，包括标题（`!!`）、列表（`*`）、链接（`[[ ]]`）、嵌入语法、宏调用等。
* 有助于模型“解析结构”，并基于语法提示理解内容的逻辑层级、关联性。
* 更适合用于：

  * 摘要/重写/结构提取
  * 转换为 markdown/html
  * 跟模型提示工程结合（比如你告诉它“解析以下 TiddlyWiki 的 wikitext，生成结构化摘要”）

### ❌ 缺点：

* 对模型来说，这种语法是“非主流语法”，没有 Markdown 那么熟悉，**必须配合提示才能准确解析**。
* 如果没有提示，大模型可能会误解格式（如将 `!!` 认为是感叹号）。

---

## ✅ 最佳实践建议（混合策略）

我的建议是：

> **优先使用方案 2（原始 wikitext）+ 精心设计的 system prompt / few-shot 提示**，但在某些场景下（如 summarization、embedding），可以选择方案 1。

### ✨ 示例提示（prompt）：

你可以给大模型加一句 system prompt，例如：

```text
你将收到 TiddlyWiki 的原始 wikitext 格式的笔记内容。请解析其结构（如标题、列表、链接），并按照层次结构理解内容。输出时可以使用 markdown 格式。
```

或更具体些：

```text
以下是一个 TiddlyWiki 的 tiddler 内容，使用 wikitext 语法。请按标题层级提取内容概要，并识别其中的超链接与项目清单。
```

---

## 🚀 更进一步（结构预处理）

你甚至可以在发送给大模型前，做一次预处理，把 wikitext 转换为结构化 JSON，例如：

```json
{
  "title": "Project Logs",
  "sections": [
    {
      "title": "Requirements",
      "items": [
        "在每个tiddler上显示一个AI对话按钮",
        "能根据tiddler的类型（使用筛选器）执行不同的动作"
      ]
    },
    {
      "title": "Project Reference",
      "items": [
        { "text": "tiddlywiki AI tools", "link": "https://github.com/..." },
        ...
      ]
    }
  ]
}
```

这种方式，最有利于精确控制输出质量。但你需要一个 wikitext parser（如 Node.js 里的 TiddlyWiki core）来实现。

---

## ✅ 总结结论

| 方案             | 理解结构性      | 表达准确性 | 适用场景             |
| -------------- | ---------- | ----- | ---------------- |
| 🅰 渲染后纯文本      | ❌ 较差       | ✅ 较高  | 简单语义摘要、embedding |
| 🅱 原始 wikitext | ✅ 很好（配合提示） | ❌ 需提示 | 高结构性任务、提纲、摘要     |
| 🅾 JSON结构      | ✅✅ 极好      | ✅✅ 极高 | 高质量输出、复杂处理任务     |

---

如果你打算开发一个像 Obsidian Copilot 那样精细的插件，那我强烈建议你**采用原始 wikitext +提示工程为主，配合局部结构化预处理的策略**。这样你的 Jarvis 插件就会具备比一般 Copilot 更强的理解能力。
